## Add mako to theme-select

**Thread:** Integrate mako (notification daemon) into the runtime theme switching system — the last program with hardcoded Catppuccin Frappe colors.

Mako's config format doesn't support imports like waybar/rofi, so used a concatenation approach: a shared `mako/base.conf` (layout/timing settings) gets concatenated with a per-theme color file from `mako/themes/` to produce the active `mako/config`. The script runs `makoctl reload` for instant application.

Key design decision during implementation: the plan had theme files concatenated before the base file (`cat theme base`), but this breaks mako's INI-like section parsing — global settings from base would end up inside the theme's `[urgency=critical]` section. Reversed to `cat base theme` and moved the critical timeout (`default-timeout=10000`) into each theme file so there's only one clean `[urgency=critical]` section at the end of the final config.

In `hyprland.nix`, replaced the full `services.mako.settings` block with just `services.mako.enable = true` plus an `mkOutOfStoreSymlink` for the config — same pattern used for zellij, helix, and yazi. Build succeeded without conflicts between the service and the symlinked config.

**Open:** Hyprlock is the only remaining program with hardcoded theme colors. All theme-select targets now covered: Alacritty, Zellij, Helix, Waybar, Rofi, Yazi, Mako.

## Add hyprlock to theme-select

**Thread:** Integrate hyprlock (lock screen) into runtime theme switching — the last program with hardcoded colors.

Hyprlock uses hyprlang, which supports `source = ./file` includes and `$variable` definitions. Used the same mutable-file pattern as waybar/rofi: `hyprlock/hyprlock.conf` sources `colors.conf`, which gets overwritten by `theme-select`. Created 7 theme files with three color variables each (`$font_color`, `$inner_color`, `$outer_color`) mapped from the theme palettes (text, overlay1, crust).

In `hyprland.nix`, replaced the full `programs.hyprlock.settings` block with `enable = true` plus `mkOutOfStoreSymlink` — same pattern as mako, zellij, helix. Also dropped `disable_loading_bar` and `no_fade_in` from general settings (not recognized by hyprlock v0.9.2). The existing hardcoded colors were Catppuccin Macchiato values, not Frappe — corrected to Frappe in the new config.

No reload mechanism needed — hyprlock reads its config fresh each time it launches, so theme changes apply on next lock.

Build succeeded with no conflicts between `programs.hyprlock.enable = true` (no settings) and the symlinked config.

**Open:** All programs now integrated into theme-select. The theme-cycling task (11) is fully complete — no remaining programs with hardcoded colors.

## Gitignore mutable theme files

**Thread:** Stop theme switches from polluting `git status` — the four mutable color files (waybar/colors.css, rofi/colors.rasi, hyprlock/colors.conf, mako/config) were tracked, so every `theme-select` run showed dirty files and risked accidental commits with non-default themes.

Replaced `cp` with `ln -sf` (relative symlinks) in `theme-select.sh` for waybar, rofi, and hyprlock — the color files are now symlinks pointing at `themes/<name>.ext` within each directory. Mako stays as concatenation (`cat base + theme > config`) since its format doesn't support imports. Added all four files to `.gitignore` and ran `git rm --cached` to untrack them while keeping them on disk.

Added a `home.activation.themeDefaults` script in `home.nix` (following the existing alacritty.nix pattern) that bootstraps missing theme files on rebuild — reads the current theme from `~/.config/alacritty/current-theme-name`, defaults to `catppuccin_frappe`, and only creates files that don't already exist. Runs after `linkGeneration` so all `mkOutOfStoreSymlink` links are in place first.

Updated `docs/theme-switching.md` with a new "Gitignored Theme Files" section and updated per-tool descriptions to reflect symlinks instead of copies.

**Open:** Changes uncommitted. User should run `theme-select catppuccin_frappe` (or current theme) to convert existing plain files into symlinks, then `nrs` to apply the activation script. Helix/Zellij were intentionally left as `sed` edits — one-line changes with minimal git noise, not worth splitting configs over.

## Hyprlock visual overhaul

**Thread:** Upgrade hyprlock from a bare-bones input field to a phone-style lock screen with clock, date, and polished input — all theme-aware.

The existing hyprlock config was minimal: screenshot background with blur and a single centered input field. Designed a clean layout with a large 80pt clock at top center, a 20pt date line below it, and a wider rounded input field slightly below screen center. Added `brightness = 0.75` to the background for contrast against varied screenshots, and text shadows on the labels for legibility.

The input field got several polish tweaks: wider (250 vs 200), rounded corners (`rounding = 15`), thinner outline (3 vs 5), no placeholder text, and state-aware colors — green check on success, red on failure with italic "try again" text, yellow on caps lock. These required three new color variables (`$accent_color`, `$fail_color`, `$warn_color`) added to all 7 theme files, each mapped to the green/red/yellow from their respective palettes.

No rebuild needed — hyprlock reads config fresh on each launch, and the theme files are already in the mutable symlink system.

## Style SDDM with Catppuccin Frappe

**Thread:** Apply Catppuccin Frappe theme to the SDDM login screen so it matches the rest of the desktop.

Planned and implemented the `catppuccin-sddm` nixpkg in `configuration.nix`. Used a `let` binding with `.override` to set flavor/accent/font (`frappe`, `mauve`, `JetBrainsMono Nerd Font`), added `theme = "catppuccin-frappe-mauve"` and `extraPackages = [ qtsvg ]` to the SDDM service block, and included the theme package in `environment.systemPackages`.

Also added `fonts.packages = [ nerd-fonts.jetbrains-mono ]` at the system level — SDDM runs as a system user before login, so it can't see home-manager fonts. NixOS deduplicates in the store so the same font in both places costs nothing.

No runtime theme switching needed — SDDM runs before login, so a fixed Frappe theme is appropriate.
